The Business Problem

Descriptive analysis answers the most fundamental question in analytics: what happened? Before diagnosing problems or predicting the future, you need to understand the baseline — the numbers, patterns, and performance across time, product, and region.

The business questions I set out to answer:

  • Which products generate the most revenue?
  • How did sales perform across different regions and categories?
  • What do monthly revenue trends look like over time?
  • What are the overall KPIs — total revenue, profit, orders, and units sold?

The dataset is a retail sales dataset spanning 2011 to 2014, containing over 9,900 transaction records across the United States — sourced from Kaggle.

Step 1 — Data Cleaning with Python

The first challenge was understanding the data properly. My initial cleaning attempt reduced the dataset from 9,994 rows to just 2,650 — a red flag that something went wrong.

The culprit was a common mistake:

❌ Wrong — drops all products except the first per order
df = df.drop_duplicates(subset=["Order ID"])

One order can contain multiple products. Deduplicating by Order ID alone deletes all but one line item per order — losing thousands of valid transactions. The correct approach removes only truly identical rows:

✅ Correct — only removes 100% identical rows
df = df.drop_duplicates()

The final cleaning script:

import pandas as pd

try:
    df = pd.read_csv("raw_data.csv", encoding="utf-8")
except UnicodeDecodeError:
    df = pd.read_csv("raw_data.csv", encoding="latin1")

# Parse dates with correct format
df["Order Date"] = pd.to_datetime(df["Order Date"], format="%Y-%m-%d", errors="coerce")
df["Ship Date"]  = pd.to_datetime(df["Ship Date"],  format="%Y-%m-%d", errors="coerce")

# Drop rows where critical fields are null
df = df.dropna(subset=["Order Date"])
df = df.dropna(subset=["Order ID", "Product ID", "Customer ID", "Sales", "Quantity", "Profit"])

# Remove only truly duplicate rows
df = df.drop_duplicates()

print(f"Total rows: {len(df)}")            # 9,994
print(f"Unique orders: {df['Order ID'].nunique()}")     # 5,009
print(f"Unique products: {df['Product ID'].nunique()}")  # 1,862

Step 2 — Star Schema Design

Rather than dumping the flat CSV directly into Power BI, I designed a proper star schema in SQL Server. This is the industry-standard approach that makes dashboards faster, cleaner, and easier to maintain.

Key rules I learned for schema design:

If you can SUM, AVG, or COUNT it to get a business insight — it's a FACT.
If it describes WHO, WHAT, WHERE, WHEN — it's a DIMENSION.
Always use the most granular, unique identifier as your foreign key. It guarantees a clean one-to-one relationship between tables.

The schema I designed — fact_sales at the center, connected to 5 dimension tables:

         dim_date
            |
dim_customer — fact_sales — dim_product
            |
       dim_location
            |
         dim_order
CREATE TABLE fact_sales (
    row_id      INT           PRIMARY KEY IDENTITY(1,1),
    order_date  DATE,
    order_id    VARCHAR(50),
    product_id  VARCHAR(50),
    customer_id VARCHAR(50),
    postal_code INT,
    sales       DECIMAL(10,2),
    quantity    INT,
    discount    DECIMAL(5,2),
    profit      DECIMAL(10,2)
);
// Star schema — fact_sales + 5 dimension tables
Important: Always use DECIMAL(10,2) for financial columns — never FLOAT. Float has rounding errors that can corrupt money calculations silently.

Step 3 — Loading Data with Python + pyodbc

With the schema ready, I wrote a Python script to extract each dimension table from the cleaned CSV and load them into SQL Server — dimensions first, fact table last.

import pandas as pd

Step 4 — Descriptive SQL Queries

Before touching Power BI, I wrote the analytical queries that would power the dashboard visuals:

-- Total Revenue and Orders
SELECT
    SUM(sales)               AS total_revenue,
    COUNT(DISTINCT order_id) AS total_orders
FROM fact_sales;

-- Top 10 Products by Revenue
SELECT TOP 10
    p.product_name,
    SUM(f.sales) AS total_revenue
FROM fact_sales f
JOIN dim_product p ON f.product_id = p.product_id
GROUP BY p.product_name
ORDER BY total_revenue DESC;

-- Sales by Region and Category
SELECT
    l.region,
    p.category,
    SUM(f.sales)  AS total_revenue,
    SUM(f.profit) AS total_profit
FROM fact_sales f
JOIN dim_location l ON f.postal_code = l.postal_code
JOIN dim_product  p ON f.product_id  = p.product_id
GROUP BY l.region, p.category
ORDER BY l.region, total_revenue DESC;

-- Monthly Revenue Trends
SELECT
    d.year,
    d.month,
    d.month_name,
    SUM(f.sales)               AS total_revenue,
    COUNT(DISTINCT f.order_id) AS total_orders
FROM fact_sales f
JOIN dim_date d ON f.order_date = d.order_date
GROUP BY d.year, d.month, d.month_name
ORDER BY d.year, d.month;

Step 5 — Power BI Dashboard

With the data warehouse and queries ready, I connected Power BI Desktop to SQL Server and built the dashboard. Power BI auto-detected most relationships. All 5 foreign key relationships were set to One-to-Many from dimension to fact table.

The one DAX measure I wrote for this page:

Profit Margin = DIVIDE(SUM(fact_sales[profit]), SUM(fact_sales[sales]))

Formatted as Percentage — gives a clean 12.47% display on the KPI card.

Key Business Insights

  • Total Revenue: $2.30M across 2011–2014 with a 12.47% profit margin
  • Top Product: Canon imageCLASS 2200 Advanced Copier at ~$60K revenue — significantly ahead of all others
  • Regional Performance: West leads, South is the weakest performing region across all categories
  • Seasonal Pattern: Sales consistently spike in Q4 (November–December) every year — classic holiday/year-end demand. Q1 is consistently the weakest period
  • Category Breakdown: Technology and Office Supplies are the dominant revenue drivers across all regions

Python + pyodbc Lessons Learned

Things I'll always do differently next time:

1. Always use format="%Y-%m-%d" for dates — not dayfirst=True
2. VARCHAR(255) for name/description columns — never guess short
3. Use .astype(int) for integer columns extracted from dates
4. dropna() on source df early — keeps all downstream tables clean
5. Add print statements before inserts — not after
6. Silent IntegrityError catch hides problems — make it verbose when debugging

What Made This Project Different

Most Superstore dataset projects on Kaggle follow a simple path: download CSV → load to Power BI → make charts. This project took a different approach — built a proper data pipeline, designed a star schema following data warehouse best practices, used SQL Server as a proper data store, and backed every visual with SQL queries for transparency and reproducibility.
